# AGENT.md — SleepLog (Local Sleep Tracker)

SleepLog is a **purely offline, on-device** sleep tracker built with **Flutter + Dart**. It has no backend, no accounts, no network calls, and no cloud storage. All data lives in a single SQLite database via **drift** on the user's phone.

**Key references:**
- Product requirements: [`docs/PRD.md`](docs/PRD.md)
- Visual identity: [`docs/brand-kit.md`](docs/brand-kit.md)

---

## 1) Core Principles

1. **Offline-first & privacy** — Zero network requests. Zero accounts. Zero analytics. The app MUST function with 100% feature parity in airplane mode. Never request unnecessary permissions (microphone, contacts, location).
2. **Type safety** — Dart sound null safety everywhere. Drift constraints validate database I/O. No `dynamic` types except where required by framework APIs.
3. **Small, reviewable changes** — Prefer incremental PRs over large rewrites. Each commit MUST leave the app in a buildable state.
4. **Test-first for core logic** — Red-Green-Refactor for repository CRUD, validation, duration calculation, and aggregation queries. Widget tests for UI components.
5. **Simplicity & YAGNI** — Only build what the PRD defines for v1. No accounts, cloud sync, wearables, sleep stages, audio recording, or smart alarms. Three similar lines are better than a premature abstraction.

## 4) Data Layer

### Drift Table Definition (`sleep_entries`)

```dart
class SleepEntries extends Table {
  TextColumn get id => text()();                          // UUID via uuid package
  TextColumn get wakeDate => text()();                    // YYYY-MM-DD (derived from wake time, local TZ)
  TextColumn get bedtimeTs => text()();                   // ISO 8601 local datetime
  TextColumn get wakeTs => text()();                      // ISO 8601 local datetime
  IntColumn get durationMinutes => integer()              // Computed on save (1–1440)
      .check(durationMinutes.isBiggerThanValue(0) & durationMinutes.isSmallerOrEqualValue(1440))();
  IntColumn get quality => integer()                      // 1–5
      .check(quality.isBiggerOrEqualValue(1) & quality.isSmallerOrEqualValue(5))();
  TextColumn get note => text().nullable()();             // Optional, max 280 chars
  TextColumn get createdAt => text()();                   // ISO 8601
  TextColumn get updatedAt => text()();                   // ISO 8601

  @override
  Set<Column> get primaryKey => {id};

  @override
  List<String> get customConstraints => [];
}
```

Indexes (defined in the `@DriftDatabase` class):
- Index on `wakeDate`
- Index on `wakeTs`

### Constraints (enforced via drift checks + Dart-level validation)

- `quality` in [1..5]
- `0 < durationMinutes <= 1440`
- `bedtimeTs` must be before `wakeTs` (within 24 hours) — validated in repository before insert/update

### Repository Pattern

- `SleepEntryRepository.create(input)` — validates input, computes `durationMinutes` and `wakeDate`, generates UUID, inserts via drift companion
- `SleepEntryRepository.list({fromDate?, toDate?, limit?, offset?})` — newest first
- `SleepEntryRepository.update(id, patch)` — re-validates, recomputes derived fields
- `SleepEntryRepository.delete(id)` — hard delete

### Drift API Notes

- Run `dart run build_runner build` after any change to table definitions or queries
- Use `NativeDatabase.memory()` for in-memory testing (no mocks needed)
- Schema migrations: increment `schemaVersion` in the database class, add migration steps in `MigrationStrategy`
- Use `Companion` objects for inserts and updates (e.g. `SleepEntriesCompanion`)

## 5) UI/UX Standards

### Component Rules

- Use **`InkWell`** or **`GestureDetector`** for tap interactions — never raw `Listener`
- Use **`ThemeData`** and the app's theme extension for all colours and text styles — never hardcode values
- Use **`SizedBox`** or `gap` (via a small helper or the `gap` package) for spacing between elements
- Use **`BorderRadius.circular(12)`** for card corners (matching `--radius-lg` from brand kit)
- Trailing commas on all widget constructor arguments for consistent formatting

### Screen States (all 4 screens)

Every screen MUST handle these 4 states, implemented via Riverpod's `AsyncValue.when()`:

1. **Loading** — skeleton or `CircularProgressIndicator`, never a blank screen
2. **Empty** — helpful message + "Log sleep" CTA (see PRD FR-011)
3. **Error** — user-friendly message with retry action
4. **Content** — the normal data-present state

### List Performance (History)

- Use `ListView.builder` (sufficient for ≤365 entries)
- Use `const` constructors on list item widgets where possible
- Extract list items into their own widget class to enable const construction

### Accessibility

- All interactive controls MUST have `Semantics` labels
- Minimum touch target: 48x48dp (Material Design) / 44x44px (Apple HIG)
- TalkBack (Android) and VoiceOver (iOS) verified for key journeys
- Follow contrast rules from the brand kit (Section 11)
- Colour MUST NOT be the sole indicator of meaning

### Brand & Visual Identity

Follow `docs/brand-kit.md` as the single source of truth. Key points:
- Default theme: **dark mode** (light mode also supported)
- Primary brand colour: `#F7931A` (orange) — CTAs, active nav indicators, focus rings
- Text on orange buttons: black `#000000` (never white)
- Typography: Satoshi (headings), Nunito (body), JetBrains Mono (monospace)
- Icons: `lucide_icons` package or Material Icons, 20x20 default size
- Spacing: 4px base grid

## 6) Testing Strategy

### Configuration

- **flutter_test** (built-in) for unit and widget tests
- Drift **in-memory database** (`NativeDatabase.memory()`) for repository/service tests — no mocking SQLite
- **mocktail** for mocking dependencies (e.g. repositories in widget tests)
- Riverpod testing: `ProviderContainer` for unit tests, `ProviderScope.overrides` for widget tests

### Test Organisation

| Type | Location | What to test |
|---|---|---|
| Unit | `test/repositories/` | Repository CRUD, constraint enforcement, duration computation |
| Unit | `test/services/` | Aggregation queries (7d, 30d), edge cases (gaps, no data) |
| Widget | `test/screens/` | Each screen renders correctly in all 4 states |
| Widget | `test/widgets/` | Shared widget behaviour and appearance |
| Integration | `integration_test/` | Key user journeys on-device |

### What MUST Be Tested

- Cross-midnight duration calculation (bedtime 23:30, wake 07:30 = 480 minutes)
- Quality range enforcement (reject 0, 6, negative, non-integer)
- Duration bounds (reject ≤0 and >1440)
- Wake date derivation from wake timestamp
- Aggregation queries with missing days / no data
- Schema migration preserves existing entries

### Run Commands

```bash
flutter test

# Specific test file
flutter test test/repositories/sleep_entry_repository_test.dart

# With coverage
flutter test --coverage

# Integration tests (requires running emulator/simulator)

# Watch mode (via build_runner — tests only)
# Re-run manually or use IDE test runner
```

## 7) Known Gotchas

Hard-won lessons — consult this before debugging.

| Topic | Gotcha | Fix |
|---|---|---|
| build_runner | Generated files (`*.g.dart`) are stale after drift/Riverpod/freezed changes | Re-run `dart run build_runner build --delete-conflicting-outputs` |
| Generated files | Editing `*.g.dart` files manually causes them to be overwritten on next build | Never edit `*.g.dart` — modify the source file and regenerate |
| Drift migrations | Schema changes without migration steps cause runtime crashes | Increment `schemaVersion` and add migration logic in `MigrationStrategy` |
| Custom fonts | Fonts not rendering despite being in `assets/fonts/` | Must register each font family + weight in `pubspec.yaml` under `flutter: fonts:` |
| fl_chart sizing | Charts render as zero-size or overflow without explicit constraints | Wrap fl_chart widgets in `SizedBox` or `AspectRatio` with explicit dimensions |
| AsyncValue.when | Forgetting to handle all 3 cases (data, loading, error) causes compile errors | Always provide all 3 callbacks: `data:`, `loading:`, `error:` |
| DateTime storage | Mixing UTC and local datetimes causes off-by-one date issues | Store local ISO 8601 strings; derive `wakeDate` from local wake time |
| intl locale | `DateFormat` defaults to en-US if `initializeDateFormatting` not called | Call `initializeDateFormatting('en_GB')` in `main()` before `runApp()` |
| Riverpod codegen | `@riverpod` annotation changes don't take effect immediately | Re-run build_runner after adding/modifying `@riverpod` annotated functions |
| go_router redirects | Redirect loops when guard logic is wrong | Test router configuration in isolation; avoid circular redirect conditions |
| Drift companions | Using the wrong companion type for insert vs update | Use `SleepEntriesCompanion.insert(...)` for new rows, `SleepEntriesCompanion(...)` for updates |
| Platform differences | fl_chart rendering differences on iOS vs Android | Test charts on both platforms early; use explicit colours (not theme-dependent for chart data) |
| pubspec.yaml | Incorrect indentation silently breaks asset/font registration | Validate YAML indentation; run `flutter pub get` and check for warnings |

## 8) Agent Workflow & Conventions

### How to Operate

When you receive a task:

1. **Restate the goal** in 1–2 sentences
2. **Read existing code** — understand patterns before modifying
3. **Make the smallest coherent change** that solves the problem
4. **Update or introduce:** types, drift schemas, tests, docs (if behaviour changes)
5. **Verify:** no network calls added, all 4 screen states handled, Semantics labels present
6. **Summarise:** files changed, how to test

### Coding Conventions

- Dart **sound null safety** — no `dynamic` types, no `!` operator unless provably safe
- File names: `snake_case.dart` (e.g. `sleep_entry_repository.dart`)
- Class/enum names: `UpperCamelCase`; variables/functions: `lowerCamelCase`
- Trailing commas on all widget constructor arguments
- Prefer `final` over `var`; use `const` constructors wherever possible
- Named exports for shared utilities
- No duplicate business logic — extract to `lib/utils/` or `lib/services/`
- Functions should be small and pure where possible
- No "magic strings" — centralise route paths, constants
- Hoist expensive constructors (`DateFormat`, `NumberFormat`, etc.) to top-level or static fields

### What NOT to Do

- **Do not** add any backend, BaaS, or cloud service (Supabase, Firebase, etc.)
- **Do not** add authentication, session management, or user accounts
- **Do not** add any library that makes network requests
- **Do not** add analytics SDKs that transmit data off-device
- **Do not** use Bloc, GetX, or Provider (we use Riverpod)
- **Do not** use `setState` in screens — use Riverpod providers for all state
- **Do not** manually edit generated `*.g.dart` files
- **Do not** add screens or navigation paths beyond the 4 defined in the PRD
- **Do not** refactor unrelated code "for cleanliness" during feature work
- **Do not** introduce new libraries without clear justification

## 10) Code Generation Workflow

Drift, Riverpod (with `riverpod_generator`), and freezed all rely on `build_runner` to produce `*.g.dart` files. This is a fundamental workflow concern.

### When to Run

Re-run code generation after any of the following changes:
- Adding or modifying drift **table definitions** or **queries**
- Adding or modifying **`@riverpod`** annotated providers
- Adding or modifying **`@freezed`** annotated classes
- Changing drift **schema version** or migration logic

### Commands

```bash
# One-off build (recommended before commit/test)
dart run build_runner build --delete-conflicting-outputs

# Watch mode (continuous rebuilds during development)
dart run build_runner watch --delete-conflicting-outputs
```

### Rules

1. **Never** edit `*.g.dart` files — they will be overwritten on next build
2. **Commit** generated files to version control (they are needed for CI and other developers)
3. **Run build_runner** before running tests or committing — stale generated files cause confusing failures
4. Use `--delete-conflicting-outputs` to avoid stale file conflicts
